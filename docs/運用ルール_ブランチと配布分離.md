# 運用ルール（ブランチと配布分離）

このドキュメントは、`reserve-LIFF` を運用する際に  
**予約-only（配布・本番）** と **もりもり版（検証・開発）** を混在させないためのルールです。

---

## 1. ブランチ戦略（固定）

- `main`: 予約-only 専用（本番・配布対象）
- `feature/full-suite`: もりもり版専用（QA、画像保存など）

### ルール

1. `main` へ直接 push しない（必ずPR）
2. もりもり版の変更は `feature/full-suite` で実施
3. `main` に戻すときは「予約-onlyに必要な差分だけ」cherry-pick か手動反映

### push 指示の運用（このプロジェクトの約束）

- あなたが「`git push`」とだけ指示した場合、**既定は `feature/full-suite` を更新**する。
- `main` への push は、**あなたの明示許可があるときだけ**実行する。
- 毎回 push 前に、次を確認する。  
  1) 現在ブランチ  
  2) push 先ブランチ  
  3) `main` 変更の有無

---

## 2. ディレクトリ分離（固定）

| 用途 | パス |
| :--- | :--- |
| 予約-only配布セット | `release/reservation-only/` |
| もりもり検証セット | `release/full-suite/` |
| 既存開発ソース | ルート配下（`app/`, `gas/`, `lib/` など） |

---

## 3. GASの分離ルール

- 予約-only正式名: `gas/Code_reservation_only.gs`
- もりもり正式名: `gas/Code_full_suite.gs`
- 互換ファイル（既存）:
  - `gas/Code_reservation_only_base.gs`（予約-only実体）
  - `gas/Code.gs`（もりもり実体）

### 配布時ルール

- お客様に渡すGASは、原則 `Code_reservation_only.gs`（実体は `Code_reservation_only_base.gs`）のみ
- もりもり版は検証契約またはオプション契約時のみ展開

---

## 4. デプロイ先分離（推奨）

| 種別 | 推奨プロジェクト名 |
| :--- | :--- |
| Vercel（予約-only） | `reserve-only-prod` |
| Vercel（もりもり） | `reserve-fullsuite-lab` |
| GAS（予約-only） | `reserve-only-gas` |
| GAS（もりもり） | `reserve-fullsuite-gas` |

---

## 5. PRチェック項目（テンプレ）

PR作成時に以下を必ず記載する。

- この変更は `main`（予約-only）に必要か？
- 変更対象は `reservation-only` / `full-suite` のどちらか？
- 本番デプロイ対象はどれか？
- 予約-only配布物に影響があるか？

---

## 6. 事故防止ルール（最重要）

1. 予約-only配布直前に、`release/reservation-only/README.md` の手順で再確認
2. もりもり機能（QA/画像保存）を `main` へ混ぜない
3. デモで全部見せても、納品物は予約-onlyを固定

---

## 7. 当面の運用手順（あなた向け）

1. まず `main` を予約-onlyとして凍結
2. `feature/full-suite` で追加開発
3. 定期的に「配布可能状態」を `release/reservation-only/` に反映
4. オプション開放時のみ `release/full-suite/` を使用

---

## 8. 現在の運用サマリ（2026-03 時点）

「GitHub は1リポジトリのまま」「Vercel は2プロジェクト分離」で運用する。

### 8-1. 運用マップ（固定）

| 区分 | 予約-only（軽量） | もりもり（フル） |
| :--- | :--- | :--- |
| GitHub Repository | `masa360/LINE-reserve` | `masa360/LINE-reserve` |
| 主ブランチ | `main` | `feature/memory-liff` |
| Vercel Project | `line-reserve` | `line-reserve-full` |
| 主な用途 | 配布・営業の基本版 | 検証・オプション提案・フル機能 |
| LINE公式アカウント | 予約-only用 | フル機能用 |
| GAS WebApp URL | 予約-only用 `/exec` | もりもり用 `/exec` |
| メモリー機能 | 使わない（メニュー非表示推奨） | 使用する（LIFFで表示） |

### 8-2. 環境変数の分離表（Vercel）

| 変数名 | 予約-only（`line-reserve`） | もりもり（`line-reserve-full`） |
| :--- | :--- | :--- |
| `GAS_WEBAPP_URL` | 予約-only GAS の `/exec` | もりもり GAS の `/exec` |
| `NEXT_PUBLIC_LIFF_ID` | 予約用 LIFF ID（予約-only公式） | 予約用 LIFF ID（フル公式） |
| `NEXT_PUBLIC_LIFF_MEMORY_ID` | （未設定でも可） | メモリー専用 LIFF ID |

> 注意: Vercel では **Key=変数名 / Value=値**。  
> 例: `NEXT_PUBLIC_LIFF_MEMORY_ID` の Value は `https://liff.line.me/...` ではなく LIFF ID 本体。

### 8-3. デプロイ運用（現在UIに合わせた手順）

1. `main` に push → `line-reserve`（予約-only）を更新  
2. `feature/memory-liff` に push → `line-reserve-full` 側でデプロイ生成  
3. `line-reserve-full` は `feature/memory-liff` のデプロイを選び、必要時に **Promote to Production**  
4. 環境変数変更後は必ず `Redeploy`

### 8-4. 日次の実務ルール（事故防止）

- 予約-onlyに出したくない機能は `main` に入れない  
- もりもり確認時は、必ず `line-reserve-full` 側でURLを確認する  
- 作業前に「どのVercelプロジェクトを触るか」を明言してから操作する  
- 迷ったら次の順で確認する:  
  1) Gitブランチ  
  2) Vercelプロジェクト名  
  3) `GAS_WEBAPP_URL` の向き先  
  4) LINE公式アカウントの対象

### 8-5. 資産所有と権限付与（当面の運用）

- 当面は、導入速度を優先して**開発者側で資産を作成**してよい  
  - LINE公式アカウント  
  - GASプロジェクト  
  - スプレッドシート  
  - 画像保存用Driveフォルダ
- 店舗へは「管理者または編集者」を付与し、利用開始後に段階移管する
- メール通知がなくても、LINE導線＋台帳管理で運用は成立する
- 権限付与時は、対象資産・権限レベル・付与日を必ず記録する
