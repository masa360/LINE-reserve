# 顧客向け 本番リリース手順書（LINE予約システム）

このドキュメントは、**美容室のお客様（導入店舗）に本番環境でリリースする**ときの実務手順をまとめたものです。  
オンライン（遠隔）前提で、トラブルを最小化する流れにしています。

---

## 1. 前提（役割分担）

| 役割 | 担当 | 内容 |
| :--- | :--- | :--- |
| 店舗オーナー/担当者 | お客様 | Google/LINEアカウントの所有、最終承認 |
| 開発・設定担当 | あなた | 実装、設定代行、テスト、切替作業 |

> 推奨: 本番資産（Googleカレンダー、GAS、Drive、LINE公式）は**店舗側が所有**し、開発者は編集権限を付与される形にする。

## 1.5 作業の切り分け（ITが苦手な美容室向け）

### 結論（先に）

- **店舗側は「承認」と「最低限のログイン操作」だけ**にする。
- **技術作業は原則あなたが担当**する。
- 迷ったら「店舗側に専門操作をさせない」を優先する。

### あなたがやる仕事（実装・設定担当）

1. 要件ヒアリングを整理し、予約-onlyの導線を設計する  
2. Next.js / GAS の実装・テストを行う  
3. Vercel本番設定（環境変数、デプロイ、動作確認）  
4. GASデプロイ、カレンダー連携、ログ連携の設定  
5. LINE Developers（Webhook、LIFF、トークン）設定  
6. リッチメニュー画像作成・タップ領域設定・リンク接続  
7. リリース前テスト（正常系/異常系）と当日立ち会い  
8. 障害時の切り戻し対応、初期サポート  

### 相手（美容室）がやる仕事（最小限）

1. Google / LINE の**オーナーアカウントを用意**する  
2. 画面共有でログインし、あなたに必要権限を付与する  
3. 店舗情報の確定値を渡す（営業時間、メニュー、価格、スタッフ名）  
4. リリース前の最終確認（文言・導線・営業時間）  
5. リリース後の一次連絡窓口（不具合報告）を決める  

### 「店舗にお願いしない」作業（IT音痴対策）

- APIキーの発行手順を口頭で丸投げする  
- GASコードの貼り付けやデプロイ操作を任せる  
- Developers / Vercel の複数画面を行き来する設定作業を任せる  
- エラー調査やログ解析を任せる  

### 店舗側に渡す依頼文テンプレ（そのまま送れる）

「技術設定はすべてこちらで行います。  
店舗様には、(1) アカウント準備、(2) ログインと権限付与、(3) 内容確認 の3点だけお願いします。  
難しい操作は不要です。画面共有で一緒に進めます。」

## 2. 本番リリースの基本方針

- 初回導入は **予約機能のみ** でリリース（最小構成）。
- QA（問い合わせ自動応答）や髪型保存は **オプション機能** として後付け。
- 本番反映前に、必ず **テスト環境で疎通確認** を完了する。

### 2.1 開発運用は「2本立て」で管理する（推奨）

**方針:**

- **見せ物・配布用:** 予約-only（説明が簡単、トラブルが少ない）
- **内部検証用:** もりもり版（予約 + QA + 保存システム）

**運用ルール（実務）:**

1. 本番配布は常に予約-onlyを基準にする  
2. もりもり版は別ブランチ/別設定で検証を進める  
3. QA・保存は契約後に機能フラグで段階的に有効化する  
4. 営業デモは予約-onlyを基本にし、必要時のみオプションを追加提示する  

**想定メリット:**

- 初期導入の障害率とサポート負荷を抑えられる
- 将来機能の開発を止めずに進められる
- アップセル（オプション提案）がしやすい

---

## 2.2 営業戦略メモ（ばらまき優先）

### 背景認識（業界トレンド）

- 「予約システムそのもの」は、無料プランの普及で**入口の無料化が標準**になっている。
- 一方で、現場が本当に課題を感じるのは、予約受付そのものよりも**運用効率・再来店・顧客管理**の部分。
- つまり、**入口は無料（または低価格）で導入しやすく**、その後の**機能価値で課金**する戦略が合理的。

### このプロジェクトでの売り方（推奨）

1. **配布フェーズ（ばらまき）**  
   - 予約-onlyを「導入しやすい基本版」として展開する。  
   - メッセージは「まずは無料〜低価格で予約導線を整える」。
2. **課題発見フェーズ**  
   - 導入店舗で、次の課題をヒアリングする:  
     - ダブルブッキング対策  
     - 問い合わせ対応の手間  
     - リピーター施策（再来店促進）  
     - 施術履歴/写真管理
3. **オプション提案フェーズ（アップセル）**  
   - QA（問い合わせ自動対応）  
   - 髪型保存（会員証・履歴）  
   - クーポン/リマインド等のCRM施策  
   を段階導入で提案する。

### 価格設計の考え方（目安）

- **基本版（予約-only）:** 入口価格を低く設定し、契約障壁を下げる
- **オプション:** 店舗の課題に紐づく価値単位で追加課金
- **全部入り:** 運用が回る店舗向けに上位プラン化

### 提案時トークの軸（短文）

- 「予約の入口は低コストでまず整える」  
- 「本当に費用対効果が出るのは、運用自動化と再来店施策」  
- 「必要なタイミングで機能を足せるので、最初に無理な投資は不要」

### 注意点（運用）

- 無料/低価格で広げる代わりに、**導入〜初期サポートを定型化**して工数を抑える。
- 予約-only配布物は固定し、個別カスタムはオプション契約で対応する。
- 「無料でどこまで対応するか」を事前に文面化し、期待値ズレを防ぐ。

## 4. 遠隔セットアップの進め方（推奨）

### パターンA（推奨）

1. お客様のアカウントにあなたを編集者/管理者として追加  
2. あなたが遠隔で設定作業を実施  
3. 最終確認をお客様に実施してもらう

### パターンB（権限付与を最小化）

1. 画面共有（Zoom/Meet）でお客様がログイン  
2. あなたが口頭で操作指示  
3. 画面共有中に設定を完了

---

## 5. 本番構築手順（予約-only）

### Step 1: GAS（予約API）を本番化

1. `Code.gs` を本番用内容で貼り付け
2. スクリプトプロパティ設定  
   - `CALENDAR_ID`  
   - `SPREADSHEET_ID`（任意）  
   - `LINE_ACCESS_TOKEN`（LINE連携時）  
   - `GAS_WEBAPP_URL` は Next.js 側で使用
3. ウェブアプリとしてデプロイ  
   - 実行ユーザー: 店舗管理者（推奨）  
   - アクセス: 本番運用に必要な公開範囲

### Step 2: Next.js / Vercel 本番反映

1. 本番ブランチへデプロイ
2. Vercel 環境変数設定（Production）  
   - `GAS_WEBAPP_URL`  
   - `NEXT_PUBLIC_LIFF_ID`
3. 再デプロイして反映確認

### Step 3: LINE設定

1. Webhook URL に GAS の `/exec` URL を設定
2. Webhook 利用をON
3. LIFF URL を確認
4. リッチメニューに予約導線を設定

## 6. リリース前テスト（必須）

### 6-1. 動作テスト

- [ ] 予約画面が開く
- [ ] メニュー選択→スタッフ→日時→確認→予約完了まで通る
- [ ] Googleカレンダーに予定が入る
- [ ] スプレッドシートにログが残る（任意）

### 6-2. 異常系テスト

- [ ] 同時間帯の重複予約が弾かれる
- [ ] 無効な入力でエラーが返る
- [ ] GAS停止時にフロントが適切に失敗表示する

### 6-3. LINE導線テスト

- [ ] トーク画面からリッチメニュー→予約画面遷移
- [ ] LIFF内でログイン状態が想定どおり

---

## 7. リリース当日の運用手順

1. 反映時間を事前に決定（営業時間外推奨）
2. 本番デプロイ実施
3. 店舗担当者と同時に最終確認
4. 初日監視（最低1営業日）
5. 問題発生時はロールバック手順に沿って戻す

---

## 8. ロールバック（切り戻し）方針

- Vercel: 前デプロイへ戻す
- GAS: 直前の安定版スクリプトへ差し戻し
- LINE: 一時的に予約導線を外す（リッチメニュー差替）
- 店舗周知: 「一時メンテナンス」文面を定型化しておく

---

## 9. セキュリティ・運用ルール（重要）

- APIキー/トークンはチャットに貼らない（安全な共有手段を使う）
- 本番アカウントの所有者は店舗側にする
- 開発者権限は最小限・不要時に解除
- 個人情報を含むデータは保持期間と削除手順を明文化

---

## 10. オプション機能の後付け導入フロー

### 10-1. QA（問い合わせ対応）

1. 予約-only本番を安定運用（目安2週間）
2. QA用シート（`ChatRules` 等）を追加
3. Gemini分類・定型回答のテスト
4. 本番ON（段階公開）

### 10-2. 髪型保存ミニアプリ

1. 保存ポリシー説明（4枚/1年 など）
2. Drive保存先・同意文言を整備
3. 画像送信～保存～閲覧導線のテスト
4. 希望店舗のみ有効化
