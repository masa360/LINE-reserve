# 要件比較メモ（予約導線 A/B）

なるほど、両方のパターンをしっかり比較・検討できるように、それぞれの**「要件定義（システムと業務の全体設計図）」**を詳細にまとめます。

システム開発を外注する際や、自分でAIに指示を出す際の「設計書」としてそのまま使えるレベルで整理しました。

---

## 【方法A】自社予約完結型（脱・ホットペッパー依存戦略）

**目的：** リピーターをLINEからの直接予約に誘導し、ホットペッパー（以下HPB）の手数料・掲載費を削減する。

### 1. システム構成（技術スタック）

- **フロントエンド:** Next.js + Tailwind CSS (Vercelデプロイ) + LIFF SDK
- **バックエンド・DB:** GAS + Googleスプレッドシート
- **スケジュール管理:** Googleカレンダー

### 2. アプリの主要機能（フロントエンド）

- **予約フロー（カレンダー連携必須）:** メニュー選択 ＞ スタッフ選択 ＞ 日時選択 ＞ 確認画面。
- **空き枠計算ロジック:** 「Googleカレンダーの空き時間」から「選択したメニューの所要時間」が連続して確保できる枠だけを画面に表示する。
- **マイページ:** 予約履歴の確認、次回の予約のキャンセル機能。

### 3. HPBとの連携ロジック（GAS）

- **HPB → LINEの自動ブロック:** お店のGmailに届くHPBの「予約確定メール」をGASが自動検知。メール本文から「日時・担当者」を抽出し、Googleカレンダーに自動で予定（ブロック）を入れる。

### 4. 現場のオペレーション（運用フロー）

- **予約が入った時:**
  - HPB経由の場合：何もしなくて良い（GASが自動でGoogleカレンダーをブロックし、LINEの枠も消える）。
  - LINE経由の場合：**【手動作業】**スタッフがHPBのサロンボードを開き、該当時間を「×（休憩など）」にして枠を閉じる。
- **メリット:** HPBのコストを大幅に削減できる。自社に顧客リストが完全に残る。
- **デメリット:** LINEから予約が入った時の「サロンボード手動ブロック」を忘れると、ダブルブッキングの事故が起きる。

## 【方法B】CRM特化・HPB送客型（業務効率＆ファン化戦略）

**目的：** 予約管理はすべてHPBに任せて現場の負担をゼロにしつつ、LINEは「会員証・ポイント・クーポン」などのファン化（LTV向上）ツールとして割り切る。

### 1. システム構成（技術スタック）

- **フロントエンド:** Next.js + Tailwind CSS (Vercelデプロイ) + LIFF SDK
- **バックエンド・DB:** GAS + Googleスプレッドシート（カレンダー連携は不要）

### 2. アプリの主要機能（フロントエンド）

- **デジタル会員証＆ポイント:** バーコード表示、現在の保有ポイントの表示。
- **クーポン一覧機能:** 「LINE限定」「お誕生日」などのクーポンを表示し、使用時に「使用済み」にできる機能。
- **予約ボタン:** タップすると、**ブラウザでHPBの自店舗の予約ページが開くだけ**。
- **来店履歴・マイカルテ:** 過去の施術内容などを表示（※データはスプレッドシートで管理）。

### 3. HPBとの連携ロジック

- **システム的な連携は一切なし。** 単純にURLのリンクでHPBへ飛ばすだけ。

### 4. 現場のオペレーション（運用フロー）

- **予約が入った時:** すべてHPB経由になるため、スタッフはこれまで通りサロンボードだけを見ていればOK。
- **会計・退店時:** **【手動作業】**お客様にLINEの会員証画面を見せてもらい、スタッフの端末（スマホ等）でQRコードを読み取ってポイントを付与する。
- **メリット:** 開発の難易度が「方法A」の半分以下になる。ダブルブッキングのリスクが完全にゼロ。現場の予約管理の負担が変わらない。
- **デメリット:** リピーターも毎回HPBを経由するため、HPBの手数料がかかり続ける。

## 比較まとめ

| 項目 | 方法A（自社予約型） | 方法B（CRM特化型） |
| :--- | :--- | :--- |
| **開発難易度** | 高い（カレンダーの空き枠計算が複雑） | 低い（画面とDBの読み書きのみ） |
| **現場の負担** | LINE予約時にHPBを手動で塞ぐ手間が発生 | 今まで通り（会計時のポイント付与のみ追加） |
| **HPBコスト** | 削減できる（リピーターの直予約化） | 削減できない（全員HPB経由） |
| **事故リスク** | スタッフの連携ミスで重複予約の恐れあり | ゼロ（すべてサロンボードで完結） |

## 配布・製品分割プラン（今後の方針メモ）

> **※伝達・計画メモ。** 実装の優先順位やファイル分割の指針として残す。

### 狙い（ビジネス）

- **まず「予約システムだけ」**を配布する。機能を絞ることで **トラブル・サポート負荷を抑える**。
- 慣れてきた店舗向けに、**後付けでオプション販売**する：
  - **チャットフローQA**（LINE Webhook ＋ スプレッドシートルール／Gemini分類など）
  - **髪型保存ミニアプリ**（画像をLINE送信→Drive保存・会員証的な履歴 等）

### 技術・リポジトリ上の分割イメージ

| パッケージ | 中身のイメージ | 備考 |
| :--- | :--- | :--- |
| **予約単体** | LIFF予約フロー ＋ GAS（カレンダー・予約API）のみ | **メイン配布・初期導入用** |
| **予約＋髪型保存** | 上記に、髪型（画像保存・ログ）系を同梱した **別ファイル／別バンドル** | 今のリポジトリ構成に近い「全部入り」版を **別名で切り出し** |
| **QA（別ファイル）** | チャットフロューQA用の **GAS または設定を別プロジェクト／別 `Code.gs` 相当** | 予約GASと **Webhook同一URLで合わせ技** も可能だが、**販売・説明は「オプション」として分離** |
| **予約＋QA 合わせ技** | 予約用GASとQA用を組み合わせた **導入キット**（説明書付き） | オプション購入者向け |

### 運用上のメリット

- 初期導入は **予約だけ** にフォーカスできる（設定項目・障害箇所が少ない）。
- QA・髪型は **必要になった店舗だけ** 追加でき、**段階的な単価アップ**がしやすい。

### 未決・後で詰めること

- 「別ファイル」の単位：**1つのGASプロジェクトを複数に分けるか**、**同一プロジェクト内で機能フラグ**にするか。
- 予約-only 配布用の **公式なZIP／ブランチ名／ドキュメント見出し** の命名。

### 実行ロードマップ（検証 → ばらまき → 拡張）

**方針の一言:** 作れる・動く見込みと **テスト運用までは QA も画像保存もやっておく**。一方で **ばらまき（配布）用として出すのは予約 LIFF だけ** に絞る。オプションは後付けで売る。

| フェーズ | やること |
| :--- | :--- |
| **1. 内部検証** | QA・画像保存も実装し、**動作確認＋テスト運用**（運用負荷・障害パターンの把握）まで実施。 |
| **2. 配布（メイン）** | **予約 LIFF のみ** を配布パッケージにする。トラブル・サポートを最小化。 |
| **3. 拡張販売** | 慣れた店舗向けに **QA / 画像保存** をオプション導入（実績ありで提案しやすい）。 |

**ブレないための最低ルール（3つ）**

1. **本番配布の中身は予約-only を固定**する（検証用に全部入りを持っていても、配布物は切り分ける）。
2. **QA／画像保存は検証環境で先に運用実績**を作ってからオプション化する。
3. **次機能を開放する条件を数値化**する（例: 2週間ノー重大障害、など）。
